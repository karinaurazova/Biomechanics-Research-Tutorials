# Interpreting failure modes

Typical segmentation failures include false bridges, broken fibers, filled pores and missing thin objects. Structural failures include noisy orientation, low coherence and wrong concentration support. Mechanical failures include parameter compensation and ill-conditioning.

Stiffness compensation is especially important: if a mask misses fibers, the solver may increase fiber stiffness to fit forces. The residual looks acceptable, but the parameter is no longer physically interpretable.

A careful conclusion should say which stage worked, under which synthetic conditions and for which metric. It should not simply say that a method works in general.

The chapter should be read together with the code. Every quantity mentioned here is created explicitly in `multimodal_benchmark.py` and regenerated by `reproduce.py`. The purpose is not to hide complexity behind a black box, but to show which assumptions enter the pipeline.

For interpretation, always ask four questions: what is the hidden truth, what is observed, what is reconstructed, and which metric compares them? This four-part logic prevents a visually appealing result from being mistaken for a verified mechanical conclusion.

## Implementation details

Failure modes can be produced deliberately by changing thresholds, noise levels or load cases. This is useful for teaching because students can see a pipeline move from success to failure and identify which metric changes first. A robust workflow should fail informatively, not silently.

## What to check while reading

Create a copy of the benchmark and remove one load case or increase polarization angle noise. Then compare the oracle and modality-based parameter scenarios. If the oracle remains accurate but modality scenarios degrade, the image-recovery stage is responsible. If the oracle also degrades, the mechanical inverse problem or observation noise is responsible.

## Typical misunderstanding

Failure is not just a negative result. In verification, controlled failure is informative. It tells you which assumptions are fragile and which experimental or computational improvements should be prioritized.
